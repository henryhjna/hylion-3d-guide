## 2. 설계 원칙

### 2.1 "어셈블을 잘 하자"

이 프로젝트는 검증된 오픈소스 컴포넌트의 통합이 핵심이다. 모든 설계 결정에서 다음 우선순위를 따른다:

1. 상용 구매 (서보, 센서, 보드, 배터리, DC-DC 컨버터)
2. 오픈소스 설계 그대로 사용 (BHL 다리, SO-ARM, IsaacLab 환경)
3. 3D 프린트로 제작 (BHL 액추에이터 기어박스, 마운트 브래킷)
4. 오픈소스 코드/모델 활용 (SmolVLA, Walking RL, Whisper, Piper TTS)
5. 최소한의 커스텀 (토르소 브래킷, 머리 조형, 오케스트레이터)

### 2.3 Sim-to-Real 전략

BHL 다리는 URDF, USD, IsaacLab 환경, sim-to-real 전이가 모두 검증되어 있다. 커스텀 상부는 시뮬에서 mass/CoM/inertia만 정확히 반영하면 되므로, 불확실성이 "상부 mass 영향"으로 한정된다. 단, 상부를 정적 mass로만 모델링하면 안 되며, 팔 스윙(precoded 및 SmolVLA)에 의한 동적 외란도 학습 환경에서 반영해야 한다. Walking RL 학습 시 상부 관절에 랜덤/주기적 토크 섭동을 주거나, 실제 팔 스윙 궤적을 재생하여 동적 외란에 대한 보행 강건성을 확보한다.

**물리 백엔드와 DGX Spark 제약:**

BHL의 공식 스택은 Isaac Lab 2.1.0 + Isaac Sim 4.5.0이며, 물리 엔진은 PhysX (Isaac Sim 기본 백엔드)이다. BHL의 sim-to-real 파이프라인(훈련 -> Sim2sim(MuJoCo) 검증 -> 실물 배포)은 모두 PhysX 기반으로 검증되어 있다.

**DGX Spark GPU 가속 불가 (확정):** DGX Spark(GB10, SM 12.1 Blackwell)에서는 PhysX GPU 가속이 지원되지 않는다. Newton 물리 백엔드도 BHL 코드와 호환되지 않아 사용 불가. 결과적으로 DGX Spark에서는 PhysX CPU 모드로 RL 훈련을 수행한다. GPU 가속 대비 훈련 속도가 느리지만, DGX Spark의 높은 CPU 코어 수와 충분한 메모리로 실용적인 훈련이 가능하다.

**Isaac Lab 3.0 마이그레이션 불필요:** Newton이 BHL과 호환되지 않으므로, Isaac Lab 3.0 포팅 작업은 진행하지 않는다. BHL 공식 스택(Isaac Lab 2.1.0 + PhysX)을 그대로 사용한다.

**시뮬 선행 원칙:** 물리 부품이 도착하기 전에 시뮬레이션, 소프트웨어, 인프라를 최대한 선행한다. 부품 도착 시 조립 -> 즉시 테스트로 직행할 수 있는 상태를 만든다.

### 2.4 듀얼 트랙 개발 구조

상체와 하체를 병렬로 개발하여 10주 내 완성 확률을 극대화한다.

```
Track A (상체): d1(리드) + e1 + e2(부분)
  -> 조작, 대화, 오케스트레이터, 상반신 통합

Track B (하체): d2(리드) + d3 + e2(부분)
  -> 다리 조립, Walking RL, sim-to-real, 보행 안정화

교차 역할:
  d3: 양쪽 AI 설계, 판단 + DGX 학습 실행 (SmolVLA 학습 + Walking RL 환경)
  e2: 양쪽 평가/검증 (SmolVLA 평가 + sim-to-real 분석 + 데이터 수집)
```

두 트랙은 Week 8에서 합류하여 상하체 결합 -> 풀 시나리오 -> Week 10 발표.

**DGX 스케줄링:** DGX Spark 슬롯 분리: Walking RL (밤 슬롯) / SmolVLA (낮 슬롯)

## 3. 시연 시나리오

발표 당일 시연의 전체 흐름. 이 시나리오가 모든 서브시스템의 요구사항을 결정한다.

```
시연 환경 레이아웃 (사전 고정):
  [관객 위치] <-~1.5m-> [로봇 홈] <-~1.5m-> [물체 테이블]
  거리, 방향은 시연 전 마커로 고정. 로봇은 홈 위치에서 시작.
```

```
1. 관객이 다가옴
   -> 로봇은 홈 위치에서 정면을 향한 채 대기 (시선 추적 없음)

2. 인사
   -> "안녕하세요" Piper TTS 재생 + 손 흔들기 (precoded)
   -> 입 서보 동기

3. 대화 -> 물체 가져오기 (fetch)
   -> 관객: "저 빨간 컵 줄 수 있어?"
   -> LLM이 fetch 태스크 트리거 -> 태스크 시퀀서 실행:
     (1) WALKING: 테이블 방향 전진 (타이머 기반, ~3초)
     (2) 정지 -> MANIPULATING: SmolVLA pick
     (3) WALKING: 홈 방향 복귀 (180도 회전 + 전진 ~3초)
     (4) 정지 -> MANIPULATING: 팔 뻗기 (precoded handover)
   -> "텀블러도 줘봐" -> 동일 시퀀스 반복
   -> "인형도 줄 수 있어?" -> 동일 시퀀스 반복

4. (보행 트랙) 자유 보행
   -> 보행 중 precoded 팔 스윙
   -> 대화 가능
```

**보행 제어 방식:** 목표 지향 보행이 아닌 타이머 기반 open-loop 보행. 시연 환경 레이아웃이 고정되어 있으므로, 보행 시간과 방향만으로 테이블~홈 간 이동이 가능하다. 비전 기반 navigation은 향후 과제(12절 참조).

**시연 시나리오 레벨:**

| 레벨 | 설명 | 조건 |
|------|------|------|
| A (풀) | 1~4 전체. 걸어가서 집고 돌아와서 전달 | 보행 + fetch 시퀀스 모두 성공 |
| B (보행 없이) | 1~3만. 물체는 팔 닿는 테이블 위, 제자리 pick-place | 보행 미달 시 |
| C (받침대) | 받침대 고정 + 1~3 | 보행 전체 실패 시 |

Week 7 말에 시나리오 레벨 예비 판정, Week 8 합류 후 최종 결정.

## 4. SmolVLA 태스크 정의

### 4.1 물체 및 수집 조건

| 물체 | 에피소드 수 | 고정 방식 |
|------|------------|-----------|
| 스타벅스 테이크아웃 컵 (빈 컵+뚜껑) | 200개 | 테이블 마커 |
| 텀블러 | 200개 | 테이블 마커 |
| 하이리온 인형 | 200개 | 테이블 마커 |

- 총 600 에피소드 (시연 조건 특화 비율 60% 이상)
- 의도적 변형 축: 마커 내 위치(+-3cm), 물체 회전(0/90/180도), 그리퍼 접근 방향(정면/측면)
- 에피소드마다 변형 조건 메타데이터 기록
- 테이블 높이, 물체 위치: Week 1 SO-ARM 작업 공간 실측 후 확정
- 시연 당일 조건(동일 테이블, 동일 조명, 동일 카메라 위치) 에피소드를 전체의 60% 이상 유지
- SmolVLA 수집/추론은 그리퍼 카메라 x2 + 외부 카메라 x1 기준 (multi-view). SmolVLA는 multi-view 입력을 네이티브 지원

### 4.2 수집 전략: 1주 집중 수집

```
Week 2: 토르소 조립 + 30개 수집 + 미니 파인튜닝 -> 수집 전략 확정 (d1 단독)
Week 3: 570개 몰아서 수집 (d1 + e2, 하루 ~120개 x 5일)
  -> 600개 x 약 8분/개(셋업, 재배치, 재시도 포함) = 80시간, 2명 분담 시 ~5일
Week 4~: d1과 e2가 머리/외장 제작으로 전환
```

수집 인수인계: d1이 Week 2에 수집 방법을 정리하고, Week 3 초에 e2와 같이 수집하면서 직접 인계.

### 4.3 2-Stage 파인튜닝 전략 (오픈소스 데이터 최대 활용)

```
Stage 1: smolvla_base (HuggingFace) 로드로 대체.
  450M 파라미터, 487 datasets, ~1천만 프레임으로 사전학습 완료된 모델.
  DGX에서 직접 사전학습 불필요.

Stage 2: 자체 600 에피소드 (메인 물체 50%, 보조 물체 25% x 2).
  균등 분배 대신 시나리오 가중 분배.
  -> Stage 1 모델에서 추가 파인튜닝 (시연 조건 특화)
```

공개 데이터가 카메라 위치와 물체가 달라도, 접근-그립-리프트 패턴을 사전 학습하면 자체 데이터 효율이 올라간다.

**W1 Dependency 체인 (블로커 순서):**
1. Day 1: Blender 스크립트 -> 외형 규격 스펙 시트 산출 (링크 치수, 관절 원점, 그리퍼 마운트 좌표+방향, FOV, 케이블 경로). 이것이 URDF, 그리퍼 카메라, 수집 파이프라인의 기준점.
2. Day 2: 스펙 시트 기반 URDF ground truth 확정 + 그리퍼 카메라 위치, 각도 확정 (설계 기준) + 작업 테이블 높이, FETCH 타이머 T1, T2 초기값 결정
3. Day 3~: smolvla_base action space -> SO-ARM101 매핑 검증 (스펙 시트 확정 후). 미검증 시 600개 수집 데이터 포맷 오염 위험.
4. W2: 팔 조립 후 그리퍼 카메라 물리 검증, 미세조정 -> 캘리브레이션 확정, 고정 (이후 변경 금지)

**SmolVLA v1 vs v2 구분:**
- v1: Stage 1 + Stage 2 기본 파인튜닝 결과. 3개 시연 물체에 대해 pick-place가 동작하는 최초 버전.
- v2: v1 이후 ablation 결과 반영 + 추가 에러 케이스 데이터 보강 + 재학습. 시연 신뢰도를 높인 최종 배포 버전.
- 전환 기준: v1이 시연 물체 3종에 대해 기본 동작(접근, 그립, 리프트)을 수행할 수 있으면 v1 완료. v2는 성공률과 안정성을 개선한 것.

### 4.4 언어 -> 동작 연결

- 온라인: Cloud LLM이 물체 종류 추출 -> SmolVLA에 전달
- 오프라인 서바이벌: Qwen 2.5 0.5B Q4 (Ollama local) + 키워드 기반 로컬 매칭, "컵" -> starbucks_cup, "텀블러" -> tumbler, "인형" -> doll

## 5. 시스템 아키텍처

**아키텍처 기준 문서: signal_flow.mermaid, signal_cables.mermaid, power_cables.mermaid, software_architecture.mermaid, state_machine.mermaid (5개 다이어그램)**

### 5.1 오케스트레이터

**기준 다이어그램: state_machine.mermaid, signal_flow.mermaid(S3f), software_architecture.mermaid(SW_ORCH)**

Orin에서 실행되는 Python asyncio 프로그램 하나가 전체 시스템을 오케스트레이션한다. 4개 입력 소스를 비동기로 리스닝하면서, 현재 상태에 따라 5개 출력 채널을 선택적으로 활성화한다.

**입력 (4개 비동기 소스):**

| 소스 | 이벤트 | 용도 |
|------|--------|------|
| Whisper (USB Mic) | VAD 감지 + 텍스트 | IDLE -> TALKING 전환 트리거 |
| Cloud/Local LLM | intent (walk/arm/fetch/talk) + 응답 텍스트 | 다음 상태 결정 |
| NUC UDP 피드백 | {state, imu_pitch, imu_roll, stable} | FETCH stable 확인, 보행 상태 모니터링 |
| NUC heartbeat 타임아웃 | UDP 응답 없음 (1초 이상) | EMERGENCY 전환 |

**출력 (5개 채널):**

| 채널 | 보내는 것 | 활성 상태 |
|------|----------|----------|
| Piper TTS -> Speaker | 응답 음성 PCM | TALKING |
| GPIO PWM | 입 서보 open/close | TALKING (TTS 재생 중) |
| UDP -> NUC | {vx, vy, vyaw} | WALKING |
| SmolVLA -> BusLinker x2 | 관절 목표 각도 | MANIPULATING |
| Camera x3 캡처 | RGB 프레임 | MANIPULATING |

**상태 정의 (4 + 1):**

```
IDLE
  입력: Whisper VAD 대기
  출력: 없음
  전환: VAD 감지 -> TALKING

TALKING
  입력: Whisper STT (GPU) + LLM 호출
  출력: Piper TTS + 입 서보
  전환: LLM intent=walk -> WALKING
        LLM intent=arm -> MANIPULATING
        LLM intent=fetch -> FETCH 시퀀스 시작
        침묵 타임아웃 -> IDLE

WALKING
  입력: NUC UDP 피드백 (상태 모니터링)
  출력: UDP vx vy vyaw + precoded 팔 스윙
  전환: 타이머 만료 또는 stop -> IDLE
        VAD 감지 -> TALKING (보행 유지하며 대화 병행)
  상호 배타: SmolVLA 비활성

MANIPULATING
  입력: Camera x3 -> SmolVLA
  출력: BusLinker x2 서보 제어
  전환: 조작 완료 -> IDLE
        실패 -> 재시도 (최대 2회) 또는 IDLE
  상호 배타: 보행 비활성

EMERGENCY
  트리거: NUC heartbeat 타임아웃 또는 물리 E-stop
  동작: Battery 1 차단 (다리 전원 OFF), 모터 토크 해제
  Orin: Battery 2 유지, 로그 보존
  어느 상태에서나 진입 가능
```

v12 대비 삭제: LOW_BATTERY 상태 (배터리 전압을 소프트웨어로 읽는 하드웨어 경로 없음, LiPo 부저가 물리 알람만 제공). FETCH 상태 (독립 상태가 아닌 오케스트레이터 시퀀스로 재분류).

**FETCH 시퀀스 (오케스트레이터가 상태 전환을 순차 구동):**

```
TALKING에서 LLM intent=fetch 수신 시, 오케스트레이터가 아래를 순차 실행:

  (1) -> WALKING: 전진, 타이머 T1초
  (2) -> 정지: UDP (0,0,0) 전송, NUC stable==true 대기 (timeout 2초)
  (3) -> MANIPULATING: SmolVLA pick (target_object)
  (4) -> WALKING: 180도 회전 + 전진, 타이머 T2초
  (5) -> 정지: NUC stable==true 대기
  (6) -> MANIPULATING: precoded handover
  (7) -> IDLE

타이머 T1, T2는 시연 환경 레이아웃에 맞춰 사전 튜닝.
실패 시: pick 실패 -> 재시도 1회 -> IDLE 복귀. stable 타임아웃 -> IDLE 복귀.
```

**stable 판정:** NUC가 BNO085 IMU pitch/roll을 모니터링하여 stable 플래그를 UDP로 Orin에 전송. 속도 명령 (0,0,0) 전송 후 pitch/roll이 임계값 이내로 수렴하면 stable=true. 임계값은 실측 기반 결정.

**Cloud LLM 장애 시:** API 호출 타임아웃 3초 -> Qwen 2.5 0.5B Q4 local fallback 자동 전환. 오케스트레이터는 LLM 소스를 모름, intent+텍스트만 수신.

**구현:** Python asyncio 단일 프로그램. asyncio.gather로 4개 입력을 병렬 대기, 상태 변수로 출력 채널 분기. ROS2나 외부 프레임워크 불필요.

### 5.2 상태별 리소스 매핑

| 상태 | Orin GPU | Orin CPU | NUC |
|------|----------|----------|-----|
| IDLE | 유휴 | 오케스트레이터 + Whisper VAD | BHL lowlevel 대기 |
| TALKING | Whisper STT | 오디오 캡처 + Piper TTS + 입 서보 PWM | BHL lowlevel 대기 |
| WALKING | Whisper STT (대화 시) | 오케스트레이터 + 오디오 캡처 | Walking policy (ONNX 250Hz) + CAN |
| MANIPULATING | SmolVLA 450M (PyTorch) | 카메라 x3 캡처 | BHL lowlevel 대기 |
| EMERGENCY | 유휴 | 로그 보존 | 정지 |

핵심 원칙:
- WALKING과 TALKING은 병행 가능 (보행 유지하며 Whisper+TTS 동작)
- WALKING과 MANIPULATING은 상호 배타 (SmolVLA 추론 중 보행 불가)
- 오케스트레이터는 모든 상태에서 Orin CPU에서 항상 실행
- 입 서보 PWM은 Orin 전담 (보행 중 NUC의 실시간성 보장)

### 5.3 컴퓨팅 분배

| 보드 | 역할 |
|------|------|
| DGX Spark | SmolVLA 파인튜닝 + Walking RL 학습 |

| NUC (BeeLink N95) | Walking RL policy (ONNX Runtime C API, 250Hz), BHL lowlevel 제어 (CAN x2, SocketCAN) |

```
[DGX Spark]
  SmolVLA 파인튜닝     Walking RL 학습
        |                    |

```

**Orin <-> NUC 통신:** Ethernet 직결, plain UDP (BHL udp_joystick.py 호환).

BHL은 ROS2를 사용하지 않는다. BHL 공식 lowlevel 코드는 plain UDP로 속도 명령을 수신하고, C 프로그램이 ONNX Runtime으로 MLP 정책을 추론한다. 이 프로젝트도 이 구조를 그대로 따른다.

| 방향 | 프로토콜 | 메시지 내용 |
|------|----------|------------|
| Orin -> NUC | UDP JSON | {mode: "walk"/"stop"/"turn", vx, vy, vyaw} -- 보행 명령 |
| NUC -> Orin | UDP JSON | {state: "standing"/"walking", imu_pitch, imu_roll, stable: bool} -- 보행 상태 피드백. FETCH stable 판정에 필수 |

**Standing policy 확인 필요:** Walking RL policy가 velocity=0 명령에서 직립을 안정적으로 유지하는지 확인해야 한다. BHL 원본 정책이 정지 상태를 지원하지 않으면, 별도 standing controller(예: PD 관절 제어)가 필요하다.

### 7.5 배터리 및 전원

**전원 기준 문서: power_cables.mermaid**

상체 CoM을 낮추기 위해 배터리를 토르소 최하단에 집중 배치. 직립 테스트 결과에 따라 Battery 1을 hip 프레임에 마운트하는 것도 검토.

| 배터리 | 대상 | 구성 | 배치 |
|:------:|------|------|------|
| Battery 1 | BHL 다리 ESC x12 -> BLDC x12 | 6S LiPo 4000mAh (22.2V nom) | 다리 프레임 |
| Battery 2 | Jetson + NUC + SO-ARM + 주변기기 | 4S LiPo 8000mAh (14.8V nom) | 토르소 최하단 |

**DC-DC 변환기 x3 (Battery 2에서 분기):**
- DC-DC #1 (buck-boost 권장): 14.8V -> 12V, 3A, NUC 전용
- DC-DC #2 (buck-boost 권장): 14.8V -> 12V, 15A, 팔 서보 전용 (BusLinker -> STS3215 x12). Battery 2 -> DC-DC #2 배선은 14AWG 이상 필수
- DC-DC #3: 14.8V -> 5V, 3A, USB Hub x2 전원

**직결:**
- Battery 1 (24V) -> ESC x12 직결 (B-G431B-ESC1 6S 정격 25.2V max)
- Battery 2 (14.8V) -> Jetson Orin Nano 직결 (9-20V 허용 범위)
- Jetson GPIO 5V -> 입 서보 SG90급 (250mA)

**LiPo 보호:** Battery 1, Battery 2 각각에 LiPo 저전압 알람(buzzer) 장착. Battery 2는 14.0V (3.5V/cell) 컷오프로 설정. DC-DC 드롭아웃 마진 확보.

**비상정지:** Battery 1 양극 직렬 NC 스위치. Battery 2는 유지 (Orin 로그 보존).

**NUC와 팔 서보 DC-DC를 분리하는 이유:** STS3215 서보 12개 동시 동작 시 순간 12A+ 전류 스파이크가 발생할 수 있어, 같은 DC-DC를 쓰면 NUC 전압이 흔들려 RL 정책 실행이 불안정해진다.

머리 제작은 7.3절 참조. 여기서는 바디와 다리 외장을 다룬다.

**바디 외장 (토르소):**
소재: 하이리온 캐릭터 컬러의 천/펠트. 고정: 벨크로. 환기구: 메쉬 소재. 구조: 프로파일 프레임 위에 천 커버를 씌우는 방식.

**다리 외장:**
다리 본체: 3D프린트 구조물 그대로 노출 (또는 얇은 EVA 커버). 발: 캐릭터 신발 느낌의 발 커버 (EVA 또는 3D프린트).

### 7.7 전원 시퀀싱

```
투입: Battery 2 ON -> Orin/NUC 부팅 -> Battery 1 ON -> BHL 캘리브레이션 -> SO-ARM 토크 ON
비상정지 후: 원인 확인 -> 안전 자세 -> Battery 1 재투입 -> IDLE 복귀
```

### 7.8 안전

팔: 토크 60%, 속도 90 deg/s, 소프트 리미트, 전류 충돌 감지, 비상정지.

보행: BNO085 IMU 데이터를 NUC에서 모니터링, IMU pitch/roll 임계값 초과 시 안전 자세(관절 토크 해제, 백드라이버블 특성으로 안전하게 주저앉음). EVA 보호대. 보행 중 SmolVLA 중단.

(v12 대비 변경: ESP32+MPU6050+MOSFET 낙상 감지 삭제. BHL은 ESP32를 사용하지 않으며, BNO085 IMU + NUC 기반 안전 로직으로 대체.)

### 7.9 열관리

Orin TDP 15W, 토르소 밀폐 시 써멀 스로틀링 위험. 40mm 팬 + 배기구(상단) + 흡기구(하단) 구조.

### 7.10 그리퍼 검증

SO-ARM101 그리퍼 jaw ~5~6cm. Week 1에서 실물체 3개로 그립 테스트. 필요 시 고무 패드 또는 물체 교체.

### 7.11 운송

상체+다리 분리 운반, 현장 조립. 퀵릴리즈 hip 구조, 10분 이내 결합.

## 8. Sim -> Hardware -> Real 파이프라인

**Walking RL 파이프라인 핵심 요소:**
- HOVER reward 구조 이식 (처음부터 설계 불필요)
- BONES-SEED + AMASS reference motion 활용
- Teacher-Student 구조: W1 중에 결정 (Teacher-Student 선택 시 ONNX export가 W6~7로 밀림)
- Walking policy -> ONNX export -> ONNX Runtime C API -> NUC 250Hz

(v12 대비 변경: "ONNX -> TensorRT -> NUC" -> "ONNX Runtime C API -> NUC". BHL은 TensorRT가 아닌 ONNX Runtime을 사용.)

```
[Week 0]  BHL IsaacLab 환경 로드 확인 (체크포인트)
[Week 1]  커스텀 상부 추가 + 파라메트릭 직립 테스트 + IsaacLab 환경 완성
[Week 2]  직립 결과 -> 체크포인트 + Walking RL 학습 시작
[Week 3]  Walking RL 학습 계속 (PhysX CPU, DGX Spark)
[Week 4]  토르소+팔 부분 실측 (시뮬 파라미터 보정용) + 다리 조립
[Week 5 초] 다리 조립 완료 -> 전체 실측 URDF 확정 + 재학습 시작
[Week 5~6] 공중 보행 -> sim-to-real 1차 비교
[Week 6]  더미 웨이트 지면 보행 -> gap 분석 + ONNX Runtime 변환 확인 (NUC C runtime) + Orin <-> NUC 통합 테스트
[Week 7]  재학습 완료 + SmolVLA v2 + 드레스 리허설
[Week 8]  실체 장착 -> 풀 시나리오 1차
[Week 9~10] 안정화 -> 발표
```

## 12. 향후 과제

### 12.1 비전 기반 Navigation

현재 보행 제어는 타이머 기반 open-loop이다. 시연 환경 레이아웃을 사전에 고정하고, 보행 시간과 방향만으로 이동한다. 이 방식은 시연에는 충분하지만, 임의 환경에서의 자율 이동은 불가능하다.

비전 기반 navigation을 이번 단계에서 하지 않는 이유:
- SLAM 또는 visual servoing이 필요. 카메라로 물체/사람 위치를 인식하고 보행을 피드백 제어하려면, 위치 추정 + 경로 계획 + 보행 속도 제어가 연동되어야 한다.
- Orin 리소스 경합. SmolVLA 추론이 GPU를 점유하고 있는 상태에서 SLAM까지 돌리면 리소스가 부족하다.
- 보행 자체가 아직 검증 중. 이족보행의 sim-to-real이 안정적으로 동작하는 것이 선결 조건.
- 10주 일정. 보행 + 조작 + 대화 통합만으로도 일정이 빡빡하다.

### 12.2 기타 향후 과제

- 목 서보 + 시선 추적 (XL430 x2 + MediaPipe, 이번 프로젝트에서 의도적으로 제외)
- 양팔 협조 조작 (현재는 단팔 pick-place만)
- 감정 표현 고도화 (음성 감정 인식 -> 표정 자동 매칭)
- 장시간 자율 운용 (배터리 자동 교체 또는 도킹 충전)
- 다중 물체 인식 및 선택 (현재는 LLM이 키워드로 물체 매칭)
