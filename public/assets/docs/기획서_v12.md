## 2. 설계 원칙

### 2.1 "어셈블을 잘 하자"

이 프로젝트는 검증된 오픈소스 컴포넌트의 통합이 핵심이다. 모든 설계 결정에서 다음 우선순위를 따른다:

1. 상용 구매 (서보, 센서, 보드, 배터리, PDB, BMS, DC-DC 컨버터)
2. 오픈소스 설계 그대로 사용 (BHL 다리, SO-ARM, IsaacLab 환경)
3. 3D 프린트로 제작 (BHL 액추에이터 기어박스, 마운트 브래킷)
4. 오픈소스 코드/모델 활용 (SmolVLA, Walking RL, MediaPipe, Whisper, ROS2)
5. 최소한의 커스텀 (토르소 브래킷, 머리 조형, 상태 머신)

### 2.3 Sim-to-Real 전략

BHL 다리는 URDF·USD, IsaacLab 환경, sim-to-real 전이가 모두 검증되어 있다. 커스텀 상부는 시뮬에서 mass/CoM/inertia만 정확히 반영하면 되므로, 불확실성이 "상부 mass 영향"으로 한정된다. 단, **상부를 정적 mass로만 모델링하면 안 되며, 팔 스윙(precoded 및 SmolVLA)에 의한 동적 외란도 학습 환경에서 반영**해야 한다. Walking RL 학습 시 상부 관절에 랜덤/주기적 토크 섭동을 주거나, 실제 팔 스윙 궤적을 재생하여 동적 외란에 대한 보행 강건성을 확보한다.

**물리 백엔드와 DGX Spark 제약:**

BHL의 공식 스택은 Isaac Lab 2.1.0 + Isaac Sim 4.5.0이며, 물리 엔진은 **PhysX** (Isaac Sim 기본 백엔드)이다. BHL의 sim-to-real 파이프라인(훈련→Sim2sim(MuJoCo) 검증→실물 배포)은 모두 PhysX 기반으로 검증되어 있다.

그러나 **DGX Spark(GB10, SM 12.1 Blackwell)에서는 PhysX GPU 가속이 지원되지 않는다.** PhysX CPU 모드로는 실용적인 RL 훈련 속도를 낼 수 없으므로, DGX Spark에서 GPU 가속 훈련을 하려면 **Newton 물리 백엔드**를 사용해야 한다. Newton은 Isaac Lab 3.0 beta(develop 브랜치)에서 제공된다. **DGX Spark에서 Newton GPU 가속이 실제 동작하는지 Week 0에 반드시 검증**해야 한다.

따라서 **BHL 코드를 Isaac Lab 3.0으로 마이그레이션하는 작업이 선행 필요**하다:
- Isaac Lab 2.x → 3.0 API 변경 (Articulation, ContactSensor 등)
- quaternion convention 변경 (wxyz → xyzw)
- Newton 프리셋 환경 설정 추가
- 첫 실행 시 JIT 컴파일 약 1시간+ 소요 (42개 커널, ~436MB 캐시, 이후 정상 속도)

**PhysX CPU 폴백 경로:** 마이그레이션 완료 전까지 DGX Spark에서 PhysX CPU 모드로 환경 검증은 가능하나 훈련 속도가 매우 느림. 실제 RL 훈련은 Newton GPU 전환 후 진행.

**시뮬 선행 원칙:** 물리 부품이 도착하기 전에 시뮬레이션·소프트웨어·인프라를 최대한 선행한다. 부품 도착 시 조립 → 즉시 테스트로 직행할 수 있는 상태를 만든다.

### 2.4 듀얼 트랙 개발 구조

상체와 하체를 병렬로 개발하여 10주 내 완성 확률을 극대화한다.

```
Track A (상체): δ1(리드) + ε1 + ε2(부분)
  → 조작, 대화, 감정, 상태 머신, 상반신 통합

Track B (하체): δ2(리드) + δ3 + ε2(부분)
  → 다리 조립, Walking RL, sim-to-real, 보행 안정화

교차 역할:
  δ3: 양쪽 AI 설계·판단 + DGX 학습 실행 (SmolVLA 학습 + Walking RL 환경)
  ε2: 양쪽 평가/검증 (SmolVLA 평가 + sim-to-real 분석 + 데이터 수집)
```

두 트랙은 **Week 8에서 합류**하여 상하체 결합 → 풀 시나리오 → Week 10 발표.

**DGX 스케줄링:** DGX Spark 슬롯 분리: Walking RL (밤 슬롯) / SmolVLA (낮 슬롯)

## 3. 시연 시나리오

발표 당일 시연의 전체 흐름. 이 시나리오가 모든 서브시스템의 요구사항을 결정한다.

```
시연 환경 레이아웃 (사전 고정):
  [관객 위치] ←~1.5m→ [로봇 홈] ←~1.5m→ [물체 테이블]
  거리·방향은 시연 전 마커로 고정. 로봇은 홈 위치에서 시작.
```

```
1. 관객이 다가옴
   → 하이리온이 시선 추적으로 바라봄 (MediaPipe → 목 PID)

2. 인사
   → "안녕하세요" TTS + 손 흔들기 (precoded — 사전 프로그래밍된 고정 관절 궤적, SmolVLA 학습 아님)
   → 감정: happy, LED 눈 + 입 서보 동기

3. 대화 → 물체 가져오기 (fetch)
   → 관객: "저 빨간 컵 줄 수 있어?"
   → LLM이 fetch 태스크 트리거 → 태스크 시퀀서 실행:
     ① WALKING: 테이블 방향 전진 (타이머 기반, ~3초)
     ② 정지 → MANIPULATING: SmolVLA pick
     ③ WALKING: 홈 방향 복귀 (180° 회전 + 전진 ~3초)
     ④ 정지 → MANIPULATING: 팔 뻗기 (precoded handover)
   → "텀블러도 줘봐" → 동일 시퀀스 반복
   → "인형도 줄 수 있어?" → 동일 시퀀스 반복

4. (보행 트랙) 자유 보행
   → 보행 중 precoded 팔 스윙
   → 대화 가능
```

**보행 제어 방식:** 목표 지향 보행이 아닌 **타이머 기반 open-loop 보행**. 시연 환경 레이아웃이 고정되어 있으므로, 보행 시간과 방향만으로 테이블~홈 간 이동이 가능하다. 비전 기반 navigation은 향후 과제(12절 참조).

**시연 시나리오 레벨:**

| 레벨 | 설명 | 조건 |
|------|------|------|
| A (풀) | 1~4 전체. 걸어가서 집고 돌아와서 전달 | 보행 + fetch 시퀀스 모두 성공 |
| B (보행 없이) | 1~3만. 물체는 팔 닿는 테이블 위, 제자리 pick-place | 보행 미달 시 |
| C (받침대) | 받침대 고정 + 1~3 | 보행 전체 실패 시 |

Week 7 말에 시나리오 레벨 예비 판정, Week 8 합류 후 최종 결정.

## 4. SmolVLA 태스크 정의

### 4.1 물체 및 수집 조건

| 물체 | 에피소드 수 | 고정 방식 |
|------|------------|-----------|
| 스타벅스 테이크아웃 컵 (빈 컵+뚜껑) | 200개 | 테이블 마커 |
| 텀블러 | 200개 | 테이블 마커 |
| 하이리온 인형 | 200개 | 테이블 마커 |

- 총 600 에피소드 (시연 조건 특화 비율 60% 이상)
- 의도적 변형 축: 마커 내 위치(±3cm), 물체 회전(0°/90°/180°), 그리퍼 접근 방향(정면/측면)
- 에피소드마다 변형 조건 메타데이터 기록
- 테이블 높이·물체 위치: Week 1 SO-ARM 작업 공간 실측 후 확정
- 시연 당일 조건(동일 테이블, 동일 조명, 동일 카메라 위치) 에피소드를 전체의 60% 이상 유지
- SmolVLA 수집/추론은 **손목 카메라** 기준 (hand-eye view). 머리 카메라는 수집에 사용하지 않음

### 4.2 수집 전략: 1주 집중 수집

```
Week 2: 토르소 조립 + 30개 수집 + 미니 파인튜닝 → 수집 전략 확정 (δ1 단독)
Week 3: ★570개 몰아서 수집★ (δ1 + ε2, 하루 ~120개 × 5일)
  → 600개 × 약 8분/개(셋업·재배치·재시도 포함) ≈ 80시간, 2명 분담 시 ~5일
Week 4~: δ1과 ε2가 머리/외장 제작으로 전환
```

수집 인수인계: δ1이 Week 2에 수집 방법을 정리하고, Week 3 초에 ε2와 같이 수집하면서 직접 인계.

### 4.3 2-Stage 파인튜닝 전략 (오픈소스 데이터 최대 활용)

```
Stage 1: `smolvla_base` (HuggingFace) 로드로 대체.
  450M 파라미터, 487 datasets, ~1천만 프레임으로 사전학습 완료된 모델.
  DGX에서 직접 사전학습 불필요.

Stage 2: 자체 600 에피소드 (메인 물체 50%, 보조 물체 25% × 2).
  균등 분배 대신 시나리오 가중 분배.
  → Stage 1 모델에서 추가 파인튜닝 (시연 조건 특화)
```

공개 데이터가 카메라 위치와 물체가 달라도, 접근-그립-리프트 패턴을 사전 학습하면 자체 데이터 효율이 올라간다.

**W1 Dependency 체인 (블로커 순서):**
1. **Day 1: Blender 스크립트 → 외형 규격 스펙 시트 산출** (링크 치수, 관절 원점, 손목 마운트 좌표+방향, FOV, 케이블 경로). 이것이 URDF, 손목캠, 수집 파이프라인의 기준점.
2. **Day 2: 스펙 시트 기반 URDF ground truth 확정 + 손목 마운트 위치·각도 확정** (설계 기준) + 작업 테이블 높이·FETCH 타이머 T1·T2 초기값 결정
3. **Day 3~: smolvla_base action space → BHL 매핑 검증** (스펙 시트 확정 후). 미검증 시 600개 수집 데이터 포맷 오염 위험.
4. **W2: 팔 조립 후 손목캠 물리 검증·미세조정 → 캘리브레이션 확정·고정** (이후 변경 금지)

**SmolVLA v1 vs v2 구분:**
- **v1**: Stage 1 + Stage 2 기본 파인튜닝 결과. 3개 시연 물체에 대해 pick-place가 동작하는 최초 버전.
- **v2**: v1 이후 ablation 결과 반영 + 추가 에러 케이스 데이터 보강 + 재학습. 시연 신뢰도를 높인 최종 배포 버전.
- **전환 기준**: v1이 시연 물체 3종에 대해 기본 동작(접근→그립→리프트)을 수행할 수 있으면 v1 완료. v2는 성공률과 안정성을 개선한 것.

### 4.4 언어→동작 연결

- 온라인: LLM이 물체 종류 추출 → SmolVLA에 전달
- 오프라인 서바이벌: 키워드 기반 로컬 매칭 (Orin) — "컵"→starbucks_cup, "텀블러"→tumbler, "인형"→doll

## 5. 시스템 아키텍처

### 5.1 상태 머신

```
상태:
  IDLE          대기. 시선 추적만 동작
  TALKING       대화 파이프라인 활성화
  MANIPULATING  SmolVLA 실행 중. 대화 요청은 완료 후 처리
  WALKING       보행 중. SmolVLA 중단, precoded 팔 스윙
  FETCH         다단계 태스크 시퀀서. 아래 서브스텝을 순차 실행
  LOW_BATTERY   SOC 20% 이하. 안전 자세, 대화만 가능
  EMERGENCY     비상정지. BHL 전원 차단, Orin 로그 유지

전환 규칙:
  TALKING 중 fetch 명령  → FETCH 진입
  FETCH 완료             → IDLE
  FETCH 중 비상정지      → EMERGENCY (현재 서브스텝 즉시 중단)
  MANIPULATING 중 대화   → 완료 후 TALKING
  어느 상태에서나 비상정지 → EMERGENCY
  어느 상태에서나 SOC 20% 이하 → LOW_BATTERY

실패 경로 (반드시 정의해야 함):
  SmolVLA pick 실패      → IDLE 복귀 또는 재시도. 실패 판정 기준과 최대 재시도 횟수를 정의할 것
  stable 대기 timeout     → IDLE 복귀. 안정화 실패 시 FETCH를 중단할 것인지 재시도할 것인지 정의할 것
  WALKING↔MANIPULATING 전환 → 팔을 안전 자세로 복귀시킨 후 보행 시작할 것. 전환 순서를 정의할 것
  LOW_BATTERY 진입 시      → SOC 측정 주체, 전달 경로, 보행 중 진입 시 정지 절차를 정의할 것
  클라우드 API 장애        → 폴백 전환 트리거 조건(타임아웃, 연속 실패 등)을 정의할 것
```

**FETCH 서브스텝 (타이머 기반 open-loop):**

```
FETCH 진입 시 LLM이 target_object 전달, 목 서보를 테이블 방향(하향)으로 전환

  ① WALKING → 테이블 방향 (전진, 타이머 T1초)
  ② 정지 → /gait/status.stable==true 확인 (안정화 대기, timeout 2초)
     → MANIPULATING (손목cam→SmolVLA pick, target_object)
  ③ WALKING → 홈 방향 (180° 회전 + 전진, 타이머 T2초)
  ④ 정지 → /gait/status.stable==true 확인
     → 목 서보를 관객 방향(정면)으로 전환
     → MANIPULATING (precoded handover)
  ⑤ → IDLE 복귀

타이머 T1, T2는 시연 환경 레이아웃에 맞춰 사전 튜닝.
서브스텝 간 전환은 상태 머신이 자동 수행.
정지→매니퓰레이션 전환 시 반드시 stable 플래그 확인 후 진입.
```

프레임워크: ε1이 Week 0에서 smach/FlexBE를 직접 평가·선정하고, Week 1 합의 미팅에서 공유.

### 5.2 상태별 프로세스-리소스 매핑

| 상태 | Orin GPU | Orin CPU | NUC | ESP32 | 네트워크 |
|------|----------|----------|-----|-------|----------|
| IDLE | 비어있음 | 머리cam→MediaPipe(CPU, 10fps) + ROS2 + 상태머신 + LED | BHL lowlevel 대기 | IMU 모니터링 | 없음 |
| TALKING | 없음 | 머리cam→MediaPipe(CPU) + 오디오 캡처 + ROS2 + TTS 재생 + LED + 입 서보 | BHL lowlevel 대기 | IMU | 클라우드 STT+LLM+TTS |
| MANIPULATING | 손목cam→SmolVLA (TensorRT) | 머리cam→MediaPipe(CPU, 2~3fps) + ROS2 + LED | BHL lowlevel 대기 | IMU | LLM 대기(선택) |
| WALKING | 없음 | 머리cam→MediaPipe(CPU) + ROS2 + LED | Walking policy + CAN write | IMU + 낙상 감지 | 대화 시 STT+LLM |

핵심 원칙:
- MediaPipe는 항상 CPU 모드 (GPU는 SmolVLA 전용)
- MANIPULATING 중에도 MediaPipe 저주파(2~3fps) 유지
- STT/LLM/TTS는 클라우드 API 사용. 로컬 fallback은 경량 STT 모델 + 키워드 매칭
- TTS 재생·LED·입 서보는 Orin 전담 (보행 중 NUC의 실시간성 보장)
- 클라우드 API 장애 시 대체 API 경로 확보 필요 (Orin CPU에서 REST 호출 → JSON 파싱 → 상태 머신 전달)

### 5.3 컴퓨팅 분배

| 보드 | 역할 |
|------|------|
| DGX Spark | SmolVLA 파인튜닝 + Walking RL 학습 |
| Orin Nano Super | SmolVLA 추론(TensorRT), MediaPipe(CPU), ROS2, 상태 머신, TTS, LED, 입 서보, SO-ARM STS3215 |
| NUC (BeeLink N95) | Walking RL policy (C 코드), BHL lowlevel 제어(CAN). BHL 공식 문서 검증 구성, 4 CAN 버스 250Hz |
| ESP32 | 낙상 감지 HW 인터럽트 → MOSFET 전원 차단 |

```
[DGX Spark]
  SmolVLA 파인튜닝     Walking RL 학습
        ↓                    ↓
  Orin (TensorRT)       NUC (BHL lowlevel C)

[Orin]                 [NUC]                [ESP32]
SmolVLA 추론      ↔   Walking policy (C)    ← IMU (낙상 전용)
MediaPipe(CPU)         BHL CAN 통신          → MOSFET 전원 차단
ROS2 + 상태 머신       IMU (policy용)
네트워크 I/O
TTS/LED/입 서보
SO-ARM STS3215
```

**Orin↔NUC 통신:** Ethernet 직결 (ROS2 토픽).

| 토픽 | 방향 | 메시지 내용 |
|------|------|------------|
| `/gait/cmd` | Orin→NUC | `{mode: "walk"/"stop"/"turn", vx, vy, vyaw}` — 보행 명령. `stop`은 제자리 직립 유지 |
| `/gait/status` | NUC→Orin | `{state: "standing"/"walking"/"fallen", vx_actual, vy_actual, imu_pitch, imu_roll, stable: bool}` — 보행 상태 피드백. `stable` 플래그로 매니퓰레이션 진입 가능 여부 판단 |

**Standing policy 확인 필요:** Walking RL policy가 velocity=0 명령에서 직립을 안정적으로 유지하는지 확인해야 한다. BHL 원본 정책이 정지 상태를 지원하지 않으면, 별도 standing controller(예: PD 관절 제어)가 필요하다. FETCH 시나리오에서 "정지→매니퓰레이션 전환"이 핵심이므로, 이 확인은 실물 테스트 이전에 시뮬에서 선행한다.

**`stable` 판정 기준:** IMU pitch/roll이 허용 범위 이내이고, `/gait/cmd stop` 이후 일정 시간 경과 후 자세 진동이 수렴한 상태. 구체적인 임계값(각도, 시간)은 실측 데이터 기반으로 결정하되, 상체팀과 하체팀이 동일한 기준을 사용해야 한다.

### 7.5 배터리 배치

상체 CoM을 낮추기 위해 배터리를 토르소 최하단에 집중 배치. 직립 테스트 결과에 따라 배터리 A를 hip 프레임에 마운트하는 것도 검토.

| 배터리 | 대상 | 구성 | 배치 |
|:------:|------|------|------|
| A | Orin + NUC + LED + 입 서보 + 스피커 | 3S2P 이상 | 토르소 최하단 또는 hip |
| B | SO-ARM STS3215 ×12 + 목 XL430×2 | 3S3P | 토르소 최하단 |
| C | BHL 다리 BLDC ×10 + ESP32 | BHL 원본 | 다리 프레임 |

전원 분배: 기성 드론용 PDB + 기성 BMS + DC-DC 벅 컨버터(5V, 12V, 19V).

비상정지: B+C 양극 직렬 NC 차단 + ESP32 MOSFET C 라인 차단. A 유지.

**정의해야 할 사항:**
- 전력 예산: SO-ARM 및 BHL BLDC의 소비 전력을 포함한 전체 전력 계산을 완성할 것
- 배터리 전압 호환: 각 배터리 구성의 출력 전압과 각 보드(Orin/NUC/서보)의 요구 전압 간 호환성을 검증할 것
- MOSFET 안전 논리: ESP32가 차단하는 범위(C만 / B+C 둘 다)를 명확히 할 것. MOSFET trip 후 복구 절차를 정의할 것
- IMU 이중화: ESP32용 IMU(낙상)와 NUC용 IMU(보행 정책)의 물리적 센서 구성과 분리 근거를 명시할 것

```
구조: 프로파일 프레임 위에 천 커버를 씌우는 방식
```

**다리 외장:**

```
다리 본체: 3D프린트 구조물 그대로 노출 (또는 얇은 EVA 커버)
발: 캐릭터 신발 느낌의 발 커버 (EVA 또는 3D프린트)
```

**외장 타임라인:**

| 시점 | 작업 | 담당 |
|------|------|------|

### 7.7 전원 시퀀싱

```
투입: A ON → Orin/NUC 부팅 → B ON → SO-ARM + 목 서보 토크 ON → C ON → BHL 캘리브레이션
비상정지 후: 원인 확인 → 안전 자세 → C 재투입 → B 재투입 → IDLE 복귀
```

### 7.8 안전

팔: 토크 60%, 속도 90°/s, 소프트 리미트, 전류 충돌 감지, 비상정지.

보행: ESP32 낙상 감지(MPU6050 → MOSFET OFF), NUC 소프트 리미트, EVA 보호대, 보행 중 SmolVLA 중단.

### 7.9 열관리

Orin TDP 25W, 토르소 밀폐 시 써멀 스로틀링 위험. 40mm 팬 + 배기구(상단) + 흡기구(하단) 구조.

### 7.10 그리퍼 검증

SO-ARM101 그리퍼 jaw ~5~6cm. Week 1에서 실물체 3개로 그립 테스트. 필요 시 고무 패드 또는 물체 교체.

### 7.11 운송

상체+다리 분리 운반, 현장 조립. 퀵릴리즈 hip 구조, 10분 이내 결합.

## 8. Sim → Hardware → Real 파이프라인

**Walking RL 파이프라인 핵심 요소:**
- HOVER reward 구조 이식 (처음부터 설계 불필요)
- BONES-SEED + AMASS reference motion 활용
- Teacher-Student 구조 선택 가능 (W2 말 결정)
- Walking policy → ONNX export → TensorRT → NUC C runtime 250Hz

```
[Week 0]  BHL IsaacLab 환경 로드 확인 (체크포인트)
[Week 1]  커스텀 상부 추가 + 파라메트릭 직립 테스트 + IsaacLab 환경 완성
[Week 2]  직립 결과 → 체크포인트 + Walking RL 학습 시작
[Week 3]  Newton 기반 Walking RL 검증 (sim-to-real 준비)
[Week 4]  실측 URDF 확정 (mass·inertia 측정) + 다리 조립
[Week 5]  공중 보행 → sim-to-real 1차 비교 + Orin TensorRT 추론 레이턴시 측정
[Week 6]  더미 웨이트 지면 보행 → gap 분석 → 재학습 루프 + ONNX → TensorRT 변환 (NUC C runtime)
[Week 7]  실측 반영 → 재학습 + SmolVLA v2 완료 + 드레스 리허설
[Week 8]  실체 장착 → 풀 시나리오 1차
[Week 9~10] 안정화 → 발표
```

## 12. 향후 과제

### 12.1 비전 기반 Navigation

현재 보행 제어는 타이머 기반 open-loop이다. 시연 환경 레이아웃을 사전에 고정하고, 보행 시간과 방향만으로 이동한다. 이 방식은 시연에는 충분하지만, 임의 환경에서의 자율 이동은 불가능하다.

비전 기반 navigation을 이번 단계에서 하지 않는 이유:

- **SLAM 또는 visual servoing이 필요.** 카메라로 물체/사람 위치를 인식하고 보행을 피드백 제어하려면, 위치 추정 + 경로 계획 + 보행 속도 제어가 연동되어야 한다. 이는 별도의 연구 주제 수준.
- **Orin 리소스 경합.** SmolVLA 추론이 GPU를 점유하고 있는 상태에서 SLAM까지 돌리면 리소스가 부족하다.
- **보행 자체가 아직 검증 중.** 이족보행의 sim-to-real이 안정적으로 동작하는 것이 선결 조건이고, 그 위에 navigation을 올리는 것은 보행이 안정화된 이후의 작업이다.
- **10주 일정.** 보행 + 조작 + 대화 통합만으로도 일정이 빡빡하며, navigation을 추가하면 다른 핵심 기능의 완성도가 떨어진다.

추후 비전 기반 navigation을 추가하면 시나리오가 "임의 환경에서 물체를 찾아 가져다주는 로봇"으로 확장된다. 이때 필요한 것:

- 경량 visual odometry 또는 depth 카메라 기반 장애물 회피
- 물체/사람 위치의 상대 좌표 추정 (MediaPipe + depth → 3D 위치)
- 보행 policy에 velocity command → waypoint following 확장
- Orin GPU 시분할 또는 추가 컴퓨팅 (Orin 2대 또는 외부 컴퓨팅)

### 12.2 기타 향후 과제

- 양팔 협조 조작 (현재는 단팔 pick-place만)
- 감정 표현 고도화 (음성 감정 인식 → 표정 자동 매칭)
- 장시간 자율 운용 (배터리 자동 교체 또는 도킹 충전)
- 다중 물체 인식 및 선택 (현재는 LLM이 키워드로 물체 매칭)
